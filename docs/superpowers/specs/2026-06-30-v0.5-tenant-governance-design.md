# v0.5 — Tenant Governance (Milestones I + J)

_Design doc. Date: 2026-06-30._

Builds on the v0.4.0 identity keystone (F: API-key auth, G: identity-aware
logging, H: per-org rate limiting). v0.5 turns identity into **governance**:
what a tenant may call, and how much it may spend.

Two milestones, one minor release (`v0.5.0`):

- **I — Model access control** — stateless, config-driven allow-lists.
- **J — Tenant budgets & quotas** — stateful, metered usage with org→team→user
  budget caps.

Sequencing: **I first** (small, low-risk), then **J** (the stateful core).

---

## Milestone I — Model access control

**Goal:** restrict which models a given key may call, returning 403 otherwise.

### Config
- Add `allowed_models: list[str]` to `KeyConfig` (default `[]`).
  - Empty/absent = **all models allowed** (backward compatible).
- `build_key_index` populates `IdentityContext.scopes` (the existing reserved
  `FrozenSet[str]`) with the allow-list. For Milestone I, `scopes` holds the set
  of permitted model ids.

### Enforcement
- In `chat_completions`, after the model is parsed: if the caller's
  `scopes` is non-empty and `model` is not in it → return **403** with the
  OpenAI error envelope, `"type": "permission_error"`.
- Only applies when auth is enabled (identity is `None` when disabled → no gate).
- `ponytail:` the check lands just after model-parse, which is after the
  rate-limit block; a denied-model request still consumes a rate token. Harmless
  for now; J revisits ordering so access denial precedes metering.

### Tests
- Allowed model → passes the gate (502 against a dead backend = passed).
- Disallowed model → 403, `permission_error`.
- Empty `allowed_models` → unrestricted.
- Auth disabled → no gate regardless of model.

### Out of scope (I)
Endpoint/method gating, named roles, per-team model policies. Model allow-list
is the single dimension.

## Milestone J — Tenant budgets & quotas

**Goal:** cap token and request usage per tenant over a period, with a full
org→team→user cascade, rejecting requests that would exceed any applicable cap.

### Metering mechanism (Approach A — pre-flight estimate)
Meter on **request count** + the **already-computed `request_ctx.cost`**
(prompt estimate × prefill_weight + max_tokens × decode_weight). Decrement
meters **before forwarding**; reject if any cap is exceeded.

- Preserves zero-copy streaming passthrough (never parse the response body).
- Gives true *pre*-rejection (blocked before hitting a backend).
- Estimate ≠ actual tokens — conservative; acceptable for budget caps.
- Future accuracy upgrade (reconcile against the upstream `usage` field) can be
  added behind the `UsageMeter` interface with no caller churn.

### `meridian/usage/` package
- **`UsageMeter` interface:**
  - `check_and_increment(keys: list[MeterKey], cost: float, requests: int=1) -> Decision`
    — atomically checks all `keys` against their caps and, if all pass,
    increments; returns allow/deny + which level/period blocked.
  - `usage(key: MeterKey) -> Usage` — current consumption (for observability).
- **Impls:**
  - `SqliteUsageMeter` (default) — embedded, survives restart. One table keyed
    by `(scope_level, scope_id, period_bucket, metric)`; atomic upsert+check in
    a transaction. SQLite file path configurable.
  - `InMemoryUsageMeter` — dict-backed, for tests / ephemeral runs.
- A third Redis impl (multi-replica) can land later without interface changes.

### Config (`BudgetConfig`)
- Per-period caps (`daily` / `monthly`) for **tokens** and **requests**,
  declarable at **org**, **team**, and **user** level.
- Per-org **rate-limit overrides** (folds in the deferred H item): a tenant may
  override the global `token_capacity` / `token_refill_rate`.
- Disabled by default (no `budgets:` block = unchanged behavior).

### Enforcement (cascade)
- A request builds the applicable meter keys: `org:{org}`, `team:{org}/{team}`
  (if team known), `user:{org}/{user}` (if user known) — for each configured
  period/metric.
- `check_and_increment` evaluates all keys; if **any** level is over cap, reject
  with **429**, `"type": "rate_limit_exceeded"`, plus `Retry-After` / reset hint
  for the binding level.
- Period rollover keyed by UTC date (`YYYY-MM-DD`) / month (`YYYY-MM`).

### Observability
- Budget rejections logged (metadata: level, period, tenant ids — same fields G
  already logs).
- Prometheus counter for rejections labeled by **level** + **period** only
  (never by tenant id — cardinality rule, CLAUDE.md §5).

### Tests
- Cascade rejection fires at each level (org / team / user) independently.
- Period rollover resets the bucket.
- SQLite meter survives a simulated restart (reopen the file).
- In-memory and SQLite impls agree on a shared behavior suite.
- Budgets disabled → no-op.

  OpenAI request/response shapes unchanged.

  section), and `README.md` features + a governance subsection as each ships.

  recipe per milestone.
